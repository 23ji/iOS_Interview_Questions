#  23. Swift의 제네릭(Generic)에 대해 설명해주세요.

## 한 줄 개념

👉 **타입을 나중에 정하는 기능**

코드를 만들 때
“Int용”, “String용” 따로 만드는 게 아니라

> “어떤 타입이든 들어올 수 있게 설계”하는 것

---

# 1️⃣ 제네릭을 왜 쓰는가?

## ❌ 제네릭이 없으면

```swift
func swapInt(_ a: inout Int, _ b: inout Int) {
    let temp = a
    a = b
    b = temp
}

func swapString(_ a: inout String, _ b: inout String) {
    let temp = a
    a = b
    b = temp
}
```

타입마다 함수 다 만들어야 함
→ 코드 중복 지옥

---

## ✅ 제네릭 사용

```swift
func swapValue<T>(_ a: inout T, _ b: inout T) {
    let temp = a
    a = b
    b = temp
}
```

`<T>` = 타입 자리 표시자

👉 Int도 되고
👉 String도 되고
👉 커스텀 타입도 됨

---

## 제네릭을 쓰는 이유 정리

* 코드 재사용성 증가
* 타입 안정성 유지
* 중복 제거
* 유연한 설계 가능

---

# 2️⃣ 제네릭 타입 파라미터란?

```swift
<T>
```

T는 그냥 이름일 뿐
Type의 약자라서 관습적으로 T 사용함

다른 이름도 가능:

```swift
<Element>
<DataType>
```

---

# 3️⃣ 제네릭은 함수뿐 아니라 타입에도 사용 가능

## 제네릭 구조체

```swift
struct Box<T> {
    var value: T
}
```

사용:

```swift
let intBox = Box(value: 10)
let stringBox = Box(value: "Hello")
```

---

## 제네릭 클래스

```swift
class Stack<T> {
    private var items: [T] = []
    
    func push(_ item: T) {
        items.append(item)
    }
    
    func pop() -> T? {
        items.popLast()
    }
}
```

# 4️⃣ 제약 조건 (Constraint)

문제 상황:

모든 타입이 다 가능한 건 아님
어떤 기능은 특정 조건이 필요함

예:

```swift
func compare<T>(_ a: T, _ b: T) -> Bool {
    return a == b   // 에러
}
```

왜?
모든 타입이 `==` 지원하지 않음

---

## 해결: 제약 조건 추가

```swift
func compare<T: Equatable>(_ a: T, _ b: T) -> Bool {
    return a == b
}
```

뜻:

👉 T는 Equatable을 따르는 타입만 가능

---

## where절 사용

```swift
func printIfEqual<T>(_ a: T, _ b: T) where T: Equatable {
    if a == b {
        print("같음")
    }
}
```

---

## 여러 제약 가능

```swift
func process<T: Equatable & Comparable>(_ value: T) {
    print(value)
}
```

---

# 5️⃣ 제네릭의 장점

### ① 타입 안정성 유지

Any 사용하면 위험

```swift
func printValue(_ value: Any)
```

→ 런타임 에러 가능성 있음

제네릭은 컴파일 타임에 타입 체크됨
→ 훨씬 안전

---

### ② 코드 중복 제거

타입별 함수 여러 개 만들 필요 없음

---

### ③ 확장성과 유지보수성 증가

라이브러리 설계 시 필수

예:

* Array<Element>
* Dictionary<Key, Value>

Swift 표준 라이브러리 대부분이 제네릭 기반

---

# 6️⃣ 주의할 점

### ① 과도한 제네릭 사용

너무 복잡해지면 가독성 나빠짐

### ② 타입 추론이 어려워질 수 있음

컴파일 에러가 복잡해짐

### ③ 모든 상황에 제네릭이 정답은 아님

공통 인터페이스가 목적이면
→ Protocol이 더 적절할 수 있음

---

# 제네릭 vs 프로토콜 차이 간단 정리

| 목적       | 제네릭           | 프로토콜       |
| -------- | ------------- | ---------- |
| 타입을 일반화  | O             | △          |
| 공통 동작 정의 | △             | O          |
| 성능       | 빠름 (컴파일 시 결정) | 약간 오버헤드 가능 |

---

# 실무에서 자주 보는 예

```swift
func fetchData<T: Decodable>(type: T.Type) -> T
```

→ 어떤 모델이든 JSON 디코딩 가능

---

# 평생 기억용 핵심 요약

* 제네릭 = 타입을 나중에 정하는 기능
* `<T>`는 타입 자리 표시자
* 코드 재사용 + 타입 안정성 확보
* `<T: Protocol>`로 제약 가능
* Swift 표준 라이브러리 대부분 제네릭 기반
